# 客户端架构
目前，从前端ui到底层通讯，客户端被分为了5层：

1. **UI层**：负责界面展示和用户交互，使用PySide6实现，处理显示以及用户交互事件。
2. **Binder层**：作为UI层和逻辑层之间的桥梁，使用信号和槽结构，将UI事件转换为逻辑层的函数调用，并将逻辑层的结果传回UI层。
3. **信息处理层**：用来处理用户输入的消息，以及服务端返回的agent聊天消息。即尝试将用户输入发送给服务器（发送方式抽象到下一层了），并将服务器返回的消息进行多媒体处理，利用binder层的信号将处理后的消息传回UI层进行展示。
4. **通讯管理层**，即network_client，用于提供与服务器交互的接口，包括登录、注册、发送消息等功能。它将UI层的请求转换为网络请求，并处理服务器的响应。
5. **websocket层**：即ws_transport，负责与服务器建立和维护websocket连接，处理实时通信。

下面介绍各个层的设计细节。

## UI层
UI层的主要实现为src/gui/main_ui.py。其核心对象是`MainWindow`。登录界面由`login_dialog.py`实现。

在`login_dialog.py`中，实现了登录界面和注册界面。登录界面包含用户名和密码输入框，自动登录勾选框，以及登录按钮和切换到注册界面的按钮。注册界面包含用户名、密码、确认密码输入框、以及邀请码输入框，以及注册按钮和切换到登录界面的按钮。

`MainWindow`包括两大部分：Live2D模型展示部分和聊天部分。

Live2D模型部分功能包括：
- 渲染Live2D模型；
- 响应用户的鼠标移动事件，改变模型的视线；
- `on_agent_thinking`函数接收binder上传的agent思考状态更新
  - 当agent进入思考状态时，显示思考气泡，循环播放思考气泡动画，每0.5秒切换到下一张；
  - 当agent退出思考状态时，隐藏思考气泡，停止动画。
- 显示背景。

聊天部分功能包括：
- 显示消息气泡，支持文本和图片消息；
  - 用户的消息气泡显示在右侧，其左侧有一块区域显示发送状态，根据binder上传的信号设置对应的状态图标（发送中、发送成功（不显示）、发送失败）； 
  - agent的消息气泡显示在左侧，其右侧有一块区域显示播放图标，点击时发送尝试播放信号，成功时更新图标；停止播放时重置播放图标。
- 工具栏，包含发送图片和调节音量
- 输入框，输入框非空时发送按钮可用。
- 响应用户的事件：
  - 用户发送文本时，向agent发送文本信息，并增加气泡；
  - 用户发送图片时，向agent发送图片信息，并增加气泡；
  - 用户打字时，向agent发送正在输入事件；
  - 用户滚动消息列表到顶部时，向binder发送加载历史消息的请求。

## 信息处理层
信息处理层主要包括`src\message_process`这个包的内容，包括两个文件`src\message_process\message_processor.py`,`src\message_process\multi_media_stream.py`。其中`message_processor.py`的核心对象是`MessageProcessor`类。其功能是处理从binder层传来的UI事件，并将需要发送的消息通过网络通讯层发送给服务器；处理从服务器发来的消息。`multi_media_stream.py`主要实现了一个`MultiMediaStream`类，用于处理agent消息中的多媒体内容（如表情和音频），并提供接口供MessageProcessor调用。

message_processor的核心架构是维护两个线程：一个是监听线程，负责监听从服务器发来的消息；另一个是发送线程，负责将需要发送的消息发送给服务器。收到的消息会被放入一个事件队列中，等待监听线程处理；需要发送的消息会被放入一个发送队列中，等待发送线程处理。

message_processor的主要接口有：
- send_typing_event：发送正在输入事件的接口，判断时间间隔是否超过预期值(0.25s)，如果超过则将事件放入发送队列。
- send_text：发送文本消息的接口，将消息放入发送队列。
- send_image：发送图片消息的接口，将消息放入发送队列。
- play_local_tts_by_uuid：根据conv_uuid请求播放本地TTS的接口，将事件放入发送队列。
  - 如果本地不存在uuid对应的音频文件，则不发送事件，返回False；
  - 把音频文件路径喂给multi_media_stream进行播放，并返回multi_media_stream的返回值，内部会进一步判断能否播放。
- stop_local_tts:停止播放当前的音频。
- set_playback_volume:设置播放音量，调用multi_media_stream的接口实现。

两个主要的线程函数：
- _listen_ws_events:监听线程的函数，从事件队列中取出消息并处理。使用`process_transport_message`具体处理每一条信息：
  - 如果信息有文本内容，则调用UI的增加回复信息的接口增加一条回复；
  - 如果信息有表情指令，则调用Live2D模型的接口更新表情；
  - 如果信息有音频内容，则调用multi_media_stream的接口播放音频。
  - 如果信息是最后一条音频，则等待音频播放结束后清理音频线程，并将音频拼装起来后保存在本地。
- _send_loop:发送线程的函数，等待发送队列中有消息需要发送时，将消息发送给服务器。
  - 具体的行为是，从队列中取出消息，并调用_send_one发送；
  - 如果成功了，利用binder的接口更新UI层的消息为submitted状态；
  - 如果失败了，并且不是终止型错误，利用binder的接口更新UI层的消息为waiting状态。并在1s后重新发送这一信息；
  - 如果失败了，并且是终止型错误，利用binder的接口更新UI层的消息为failed状态。

multi_media_stream主要的接口和行为包括：
- feed_local_wav：喂入本地wav音频文件的接口，
  - 如果当前正在播放服务端内容，或音频不存在，返回False
  - 如果当前正在播放别的本地内容，先停止当前播放，再喂入新的音频文件，并开始播放，返回True
- feed：喂入服务端音频内容的接口，会打断本地播放，并喂入服务端音频。
  - 所有的服务端音频均为wav格式，只有第一个包有包头；
- set_volume_percent：更新音量的接口，参数为0-100的整数。

## 通讯管理层
主要实现是`src\network\network_client.py`中的`NetworkClient`类。登录相关的依赖`auth.py`提供的接口，使用POST请求。聊天相关的依赖`ws_transport.py`提供的接口包括：
- login:登录接口，发送登录请求并处理响应。
- register:注册接口，发送注册请求并处理响应。
- auto_login:自动登录接口，发送自动登录请求并处理响应。
- get_history:获取历史消息接口，发送获取历史消息请求并处理响应。
  - 如果获取的历史信息包含图片，而且本地没有，会下载图片并保存在本地。然后更新图片对应的本地路径。
- send_chat：发送文本聊天消息接口；返回WS信息发送结果（一个字典，包括ok, request_id, error, drop字段，drop字段表示是否放弃重新发送）
- send_image：发送图片聊天消息接口；
- send_typing_event：发送正在输入事件接口。

## 5. websocket层
主要实现是`src\network\ws_transport.py`中的`WebSocketTransport`类。其功能是提供一个与服务器建立和维护websocket连接的接口，并处理实时通信。其核心功能包括：
1. 建立WS连接，根据协议规定进行鉴权；
2. 提供发送消息的接口，包括发送文本消息、图片消息、正在输入事件等；
3. 发送后，接收ACK消息，并将结果返回给调用者；
4. 发送心跳包，间隔为10s，保持连接活跃；
5. 监听服务器发来的消息，将` WSEventType.AGENT_STATE_CHANGED`和`WSEventType.AGENT_MESSAGE`类型的消息利用回调函数交由上层处理；
6. 监听服务器发来的错误，创建一个虚拟的agent回复，展示错误信息。
7. 连接中断后尝试重连。

        if not base_url.startswith("https://"):
            return None
        if self.verify_ssl:
            ctx = ssl.create_default_context()
        else:
            ctx = ssl._create_unverified_context()

        # Improve compatibility with some tunneling endpoints in this project runtime.
        ctx.minimum_version = ssl.TLSVersion.TLSv1_2
        return ctx
