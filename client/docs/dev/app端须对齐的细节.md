# app端须对齐的细节
本文档将为APP客户端的升级提供一些设计上和功能上的约束。

已知当前客户端被划分为5个层级：
1. **UI层**：负责界面展示和用户交互，使用PySide6实现，处理显示以及用户交互事件。
2. **Binder层**：作为UI层和逻辑层之间的桥梁，使用信号和槽结构，将UI事件转换为逻辑层的函数调用，并将逻辑层的结果传回UI层。
3. **信息处理层**：用来处理用户输入的消息，以及服务端返回的agent聊天消息。即尝试将用户输入发送给服务器（发送方式抽象到下一层了），并将服务器返回的消息进行多媒体处理，利用binder层的信号将处理后的消息传回UI层进行展示。
4. **通讯管理层**，即network_client，用于提供与服务器交互的接口，包括登录、注册、发送消息等功能。它将UI层的请求转换为网络请求，并处理服务器的响应。
5. **websocket层**：即ws_transport，负责与服务器建立和维护websocket连接，处理实时通信。

我们分别从五个层级来说明需要对齐的细节。五个层级包含的功能见`客户端架构.md`文档。

## 1.UI层
在登陆界面部分，注册界面需要增加一栏“确认密码”，并且需要对两次输入的密码进行校验，确保一致后才能发送注册请求。若不一致在本地弹出提示，要求用户修改。

在聊天界面部分，Live2D区域需要增加对thinking的处理，具体包括：
- 当agent进入思考状态时，显示思考气泡，循环播放思考气泡动画，每0.5秒切换到下一张；
- 当agent退出思考状态时，隐藏思考气泡，停止动画。
- thinking气泡的位置必须要在Live2D模型图层之上，在Live2D区域的相对位置和PC端保持一致。

聊天部分功能包括：
- 移除用户发送数据之后，等待agent回复才能继续输入的限制，改为用户发送数据之后可以继续输入和发送数据。
- 增加用户正在输入的状态提示。当用户在输入框内输入内容时，向服务端发送`user_typing`事件，告知用户正在输入。
- 移除用户发送数据之后，客户端自动生成一个新的等待气泡，等agent回复之后再移除这个等待气泡的功能。当前等待的UI设计移动到Live2D区域，并且由服务端发来的agent状态来控制显示和隐藏。
- PC版本客户端加入了音量控制功能，APP端**不需要**这个功能。
- 在气泡的侧面增加一个label，
  - 对用户消息而言在其左侧，表示发送状态（发送中、发送成功、发送失败）；
  - 对agent消息而言在其右侧，表示播放，点击可以触发尝试播放本地语音的功能，播放时再次点击会停止播放。接收到停止播放信号时重置图标。
  - 当前播放的是哪一个音频，以及点击其他气泡时如何修改当前播放音频的状态，需要一个独立的对象统一管理所有气泡。

## 2.Binder层
在APP端需要抽象出明确的binder层作为前端和底层通讯的桥梁，binder层的功能介绍参见`Binder层`部分。

## 3. 信息处理层
APP的信息处理层需要增加
- 保存音频到本地的功能；
- 维护两个播放队列，
  - 服务端发来的放入队列A，这个队列会同步移动口型，其行为与当前一致；
  - 用户点击播放时放入队列B，如果当前有正在播放的，停止原来的并播放新的；如果接收到停止播放的信号需要停止；如果队列A为空，不尝试播放；如果播放途中接收到新的消息也直接停止。这是需要新增的。
- 发送typing事件的功能，需要根据当前队列是否有消息，以及是否已经过了冷却时间来决定是否发送typing事件。
- 维护两个消息队列，从队列里面拿消息播放，和从队列里拿消息发送；
- 发送消息后，根据是否发送成功判断是否重试，并将发送结构借由binder发回UI。

注意，信息处理层处理了音量事件，APP端不需要这个功能，所以相关代码可以不实现。

## 4. 通讯管理层
通讯管理层是继承了原有的网络通讯功能，但是把聊天相关的接口修改为调用Websocket层封装的接口。

由于信息处理层发送之后需要等待一个返回值来判断是否发送成功，因此通讯管理层在实现`send_chat`, `send_image`, `send_typing`等接口时需要返回一个发送结果。其他接口的实现几乎和当前实现一致，具体以代码为准。

## 5. websocket层
websocket为本次更新全新的技术栈。在实现内容上和`客户端架构.md`中描述的一致，具体以代码为准。

主要实现是`src\network\ws_transport.py`中的`WebSocketTransport`类。其功能是提供一个与服务器建立和维护websocket连接的接口，并处理实时通信。其核心功能包括：
1. 建立WS连接，根据协议规定进行鉴权；
2. 提供发送消息的接口，包括发送文本消息、图片消息、正在输入事件等；
3. 发送后，接收ACK消息，并将结果返回给调用者；
4. 发送心跳包，间隔为10s，保持连接活跃；
5. 监听服务器发来的消息，将` WSEventType.AGENT_STATE_CHANGED`和`WSEventType.AGENT_MESSAGE`类型的消息利用回调函数交由上层处理；
6. 监听服务器发来的错误，创建一个虚拟的agent回复，展示错误信息。
7. 连接中断后尝试重连。
   
注意，为保证兼容性，PC端有如下实现：
```Python
    @staticmethod
    def _build_ws_url(base_url: str) -> str:
        if base_url.startswith("https://"):
            return "wss://" + base_url[len("https://") :].rstrip("/") + "/chat_ws"
        if base_url.startswith("http://"):
            return "ws://" + base_url[len("http://") :].rstrip("/") + "/chat_ws"
        raise ValueError("base_url must start with http:// or https://")

    def _build_ssl_context(self, base_url: str):
        if not base_url.startswith("https://"):
            return None
        if self.verify_ssl:
            ctx = ssl.create_default_context()
        else:
            ctx = ssl._create_unverified_context()

        # Improve compatibility with some tunneling endpoints in this project runtime.
        ctx.minimum_version = ssl.TLSVersion.TLSv1_2
        return ctx
```
