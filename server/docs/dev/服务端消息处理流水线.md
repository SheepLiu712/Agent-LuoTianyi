# 服务端消息处理流水线
## 处理需求
考虑用户会在任意时刻发送任意条消息（这里特指聊天消息，不包括通讯控制指令）过来。

我们希望用户的每条消息都能被及时处理，不要让用户等待太久。

我们要求，整个调用链路中，LLM的调用次数不超过2次，对于图片信息可以额外多一次。其他的处理都应该是轻量级的。

## 方案
### 总览
我们维护一个unread_user_messages列表，储存用户发送过来但还未被处理的消息。我们通过LLM可以处理这个列表内的信息，它总会分为两部分：
- 组成了完整话题的消息（比如用户打了招呼，完整地提了个问题，或者分享了自己的一段经历），称为完整话题消息；
- 不组成完整话题的消息（比如用户明显在讲一个话题，但还没有说完），称为不完整话题消息。

在非常理想的条件下（但也不是很强的条件），我们认为这个列表内，完整话题消息总是一个前缀，后面跟着不完整话题消息。

那么对于这个列表的处理，我们可以采用如下策略：
- 用LLM从中提取完整话题，构成一个话题列表；
- 考虑话题的覆盖关系，例如一个打招呼话题和一个提问话题，后者覆盖前者，我们就只保留提问话题；
- 对于每个话题，我们都用LLM进行处理，得到一个Agent的回复；
- 处理完一个话题后，我们就把它从unread_user_messages里删除
- 对于不完整话题信息，我们等用户说完了再处理。如果用户一直没有说完，等待超时的时候，我们直接把它当成一个完整话题来处理。

得到话题之后，我们需要搜索记忆，组装提示词，调用LLM得到回复。进而，将回复进行TTS，然后发回用户。

### 工作阶段
我们按照一条消息的处理流程，可以把它划分为如下几个阶段：
- 前处理阶段，发生在消息到达服务端时：
  - 如果是图片消息，首先调用VLM做一次处理，得到图片的描述信息
  - 利用关键词识别其中的术语，将术语附带到LLM处理的上下文中。这一步暂时用来做歌词识别，即判断用户发的消息是不是歌词。
  - 将聊天消息入库；
  - 将消息添加到unread_user_messages中。
- 话题提取阶段，每次有一个消息进入unread_user_messages时：
  - 调用LLM对unread_user_messages进行话题提取，得到一个话题列表；
  - 对话题列表进行覆盖关系的处理，得到最终的话题列表；
  - 更新不完整话题消息的倒计时；
  - 如果不完整话题消息的倒计时到了，就把它当成一个完整话题来处理。
- 记忆检索与回复阶段，每有一个话题，按照话题出现的先后顺序处理：
  - 根据话题内容，搜索相关的向量记忆；
  - 如果话题有事实约束（例如是否涉及某一首洛天依的歌），检索相关知识库记忆。
  - 组装提示词，调用LLM得到回复。
- 多媒体回复阶段：
  - 将LLM的回复进行TTS，得到语音数据；
  - 将文本回复和语音数据发回用户。
- 后处理阶段：
  - 将Agent的回复进行入库；这一步在Agent生成回复之后就可以做了，不需要等到多媒体回复阶段完成。
  - 更新向量记忆库。这一步在话题生成之后就可以做了，不需要等到回复生成之后。
  - 压缩上下文。在回复生成之后就可以做了，不需要等到多媒体回复阶段完成。
  - 在充分多的消息之后，进行一次用户人设的更新。这一步的触发条件和更新策略需要设计一下。

### 细节
在`chat_stream.py`中实现流水线，在`luotianyi_agent.py`中实现具体的处理逻辑。

可能需要使用多个生产者-消费者队列来实现上述的处理流程，即：
- 前处理阶段完成后将消息放入unread_user_messages， 由话题提取模块消费；
- 话题提取模块将话题放入一个话题队列，由回复生成器消费；
- 回复生成器检索知识，生成回复后进入多媒体回复阶段。
- 多媒体回复阶段生成音频，将回复发回用户。目前这就是`global_speaking_worker`的功能了。

#### 前处理阶段
- 对于图片消息的处理，目前旧的架构中有对应的接口，可以尝试直接调用。
- 先进行关键词识别，将识别出的术语附带到LLM处理的上下文中。
- 入库对应`conversation_manager.add_conversation()`接口，可以尝试直接调用。
- 将消息添加到unread_user_messages中。这个队列需要保存在chat_stream中，Redis中还是db中，需要考虑。

#### 话题提取阶段
- 话题提取的LLM调用需要设计好提示词，确保它能正确地将消息分为完整话题和不完整话题两类。
- 提取出来的话题应该包括以下内容：
  - topic_type：话题类型，包括打招呼、提问、分享经历、回复上文、求安慰、其他聊天；
  - topic_content：话题的内容，例如用户打了什么招呼，提了什么问题，分享了什么经历等；
  - topic_keywords：话题包含的关键词，用于后续的向量检索；
  - fact_constraints：话题中包含的事实约束，例如是否涉及某一首洛天依的歌等。
- 打招呼话题可以被任何其他话题覆盖，其他都不会覆盖。覆盖逻辑手动处理，不依赖LLM。
- 需要处理话题结束定时器。它只决定当前剩下的不完整话题信息是否被当成一个完整话题来处理，不影响已经提取出来的话题。
  - 当接收到新的消息的时候，如果当前有不完整话题信息，我们就重置它的定时器。
  - 当接收到typing事件的时候，如果当前有不完整话题信息，我们也重置它的定时器。
  - 定时器的时长设置为2s即可。
  - 当定时器到期的时候，我们就把当前剩下的不完整话题信息当成一个完整话题来处理。
  
#### 记忆检索与回复阶段
- 根据话题内容中的关键词进行向量检索，得到相关的记忆片段。
- 根据话题内容中的事实约束进行知识库检索，得到相关的记忆片段。
- 生成的提示词应该包括以下内容：
  - 天依的人设信息；
  - 用户的人设信息，包括用户的身份、职业，和天依的关系。
  - 部分上下文。包括40句最近对话，以及长程讨论话题（可能没有）
  - 话题内容，即topic_content
  - 相关的记忆片段，包括向量记忆和知识库记忆
- LLM的回复可以以自然语言回复，之后再切分句子进行TTS；也可以直接让LLM输出一个结构化的回复，包含文本内容和语音内容两部分。需要设计好提示词，确保LLM能正确地按照要求输出。
  - 在当前的架构中使用了结构化的回复，优点是方便处理，缺点是没法流式输出。可以先实现结构化回复，后续如果需要流式输出再进行改造。
  - 如果使用自然语言回复，那就需要
    - 对于live2d表情和TTS参考音频的控制，需要做情感识别；
    - 如果有唱歌需求，要要求LLM在回复中注入唱歌指令。

#### 多媒体回复阶段
- 将LLM的回复进行TTS，得到语音数据。这部分倒是很明确。

#### 后处理阶段
- 将Agent的回复进行入库。这个接口也是add_conversation，可以直接调用。
- 更新向量记忆库。注意，现在的向量记忆库还提供了修改记忆的功能，结果deepseek把记忆越写越长。
  - 我们需要明确记忆的表述就是一句话，复杂的记忆分成多条；
  - 我们需要明确记忆的内容：1.用户画像，包括其经历、喜好等；2.用户和天依讨论的话题，做的事情；
  - 记忆写了就写了，不需要修改了。
- 压缩上下文。在回复生成之后就可以做了，不需要等到多媒体回复阶段完成。
  - 对于被截断的上下文，可以让LLM做压缩。只保留贯穿到现在的长程话题，其他内容不用压缩。
- 人设更新，可以维护一个计数器，当消息数量达到一定数量（比如100条）之后，就触发一次人设更新。
  - 人设更新的内容包括用户画像和用户与天依的关系等，可以让LLM根据之前的对话内容进行总结和更新。
