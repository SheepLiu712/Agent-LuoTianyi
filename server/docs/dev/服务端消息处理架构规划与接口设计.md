# 服务端消息处理架构规划与接口设计

## 1. 目标与约束

### 1.1 业务目标
- 用户在任意时刻发送任意条聊天消息时，都能被及时处理。
- 优先降低首包等待时间，避免“卡住不回”。
- 支持文本与图片输入，支持 typing 事件驱动的话题补全。

### 1.2 主链路调用预算
- 文本消息主链路 LLM 调用次数严格为 2 次：
  - 第 1 次：话题提取（Topic Extraction）
  - 第 2 次：回复生成（Reply Generation）
- 图片消息允许额外 1 次 VLM 调用（用于图片描述）。
- 记忆写入、上下文压缩、人设更新属于后处理，不计入主链路预算。

### 1.3 已确认设计决策
- 部署形态：多实例优先。
- unread_user_messages：会话内存存储。
- 回复输出：先采用结构化整包回复。
- 调用预算：主链路 2 次 LLM 为硬约束。

---

## 2. 总体架构

### 2.1 分层结构
1. Ingress 预处理层
- 解析 USER_TEXT、USER_IMAGE、USER_TYPING。
- 图片先做 VLM 描述。
- 术语识别（如歌词线索）并附加到消息上下文。
- 生成标准化 UnreadMessage，写入 unread 缓冲。

2. Topic Extraction 话题提取层
- 对 unread 批次做话题切分，输出：
  - complete_topics（完整话题）
  - incomplete_tail（不完整尾部）
- 应用覆盖规则：greeting 可被其他类型覆盖。
- 维护 2 秒不完整话题计时器（text 与 typing 均重置）。

3. Reply Generation 回复生成层
- 按话题时间顺序消费（每用户 FIFO）。
- 进行向量记忆检索与知识检索。
- 组装提示词并调用 LLM 生成结构化回复。

4. Media Output 多媒体输出层
- 将结构化回复转为 TTS 音频。
- 统一封装 ChatResponse 并通过 WebSocket 回发。
- TTS 失败时降级为纯文本发送。

5. Post Process 后处理层（异步）
- Agent 回复入库。
- 向量记忆写入。
- 上下文压缩。
- 人设更新计数与触发。

### 2.2 多实例一致性前提
- 由于 unread 存在内存，接入层必须按 user_id 做粘性路由。
- 同一 user_id 的消息必须落到同一实例，才能保证会话顺序与状态一致。
- 若未来无法保证粘性路由，需引入 unread 存储抽象并切换 Redis 实现。

## 3. 领域对象设计

## 3.1 UnreadMessage
- message_id: string
- user_id: string
- message_type: enum(text, image)
- raw_text: string
- image_url: string | null
- image_caption: string | null
- keyword_tags: list[string]
- created_at_ms: int

## 3.2 ExtractedTopic
- topic_id: string
- topic_type: enum(greeting, question, share, reply_context, comfort, other)
- topic_content: string
- topic_keywords: list[string]
- fact_constraints: list[string]
- source_message_ids: list[string]
- is_forced_from_incomplete: bool

## 3.3 TopicBatch
- complete_topics: list[ExtractedTopic]
- incomplete_tail_messages: list[UnreadMessage]
- extraction_meta:
  - model_name: string
  - prompt_version: string
  - elapsed_ms: int

## 3.4 ReplyPlan
- topic_id: string
- utterances: list[OneSentenceChat | SongSegmentChat]
- memory_hits: list[string]
- knowledge_hits: list[string]
- reply_meta:
  - model_name: string
  - elapsed_ms: int
  - token_usage: object

## 3.5 PostProcessTask
- task_id: string
- user_id: string
- conversation_items: list[object]
- do_memory_write: bool
- do_context_compress: bool
- do_persona_counter_inc: bool

## 4. 接口级设计

## 4.1 预处理接口

### PipelineIngress.accept_event
- 输入：
  - user_id: string
  - event: ChatInputEvent
- 输出：PreprocessResult
  - normalized_messages: list[UnreadMessage]
  - should_reset_incomplete_timer: bool
  - has_new_content: bool
- 行为：
  - USER_TEXT: 直接标准化。
  - USER_IMAGE: 先 VLM，再标准化。
  - USER_TYPING: 不入 unread，仅重置计时器。

## 4.2 unread 缓冲接口

### UnreadStore.append
- 输入：user_id, messages
- 输出：none

### UnreadStore.snapshot
- 输入：user_id
- 输出：UnreadSnapshot
  - snapshot_id: string
  - messages: list[UnreadMessage]

### UnreadStore.ack_processed
- 输入：user_id, snapshot_id
- 输出：none
- 说明：采用“快照处理成功后确认删除”，避免异常导致消息丢失。

### UnreadStore.rollback
- 输入：user_id, snapshot_id
- 输出：none
- 说明：话题提取失败时恢复快照。

## 4.3 话题提取接口（主链 LLM #1）

### TopicExtractor.extract
- 输入：
  - user_id: string
  - snapshot: UnreadSnapshot
  - short_context: string
- 输出：TopicBatch
- 约束：
  - 一次调用返回完整话题与不完整尾部。
  - 输出字段必须完整（topic_type/topic_content/topic_keywords/fact_constraints）。

### TopicCoverageResolver.resolve
- 输入：list[ExtractedTopic]
- 输出：list[ExtractedTopic]
- 规则：
  - greeting 被任意非 greeting 覆盖。
  - 其他类型不互相覆盖。

## 4.4 不完整话题计时接口

### IncompleteTopicTimer.reset
- 输入：user_id
- 输出：none

### IncompleteTopicTimer.on_timeout
- 输入：user_id
- 输出：ForcedTopicEvent
  - forced_topic: ExtractedTopic
- 说明：2 秒超时将 incomplete_tail 强制转完整话题。

## 4.5 回复生成接口（主链 LLM #2）

### ReplyComposer.compose
- 输入：
  - user_id: string
  - topic: ExtractedTopic
  - context_window: string
- 输出：ReplyPlan
- 内部步骤：
  - memory_search（向量记忆）
  - knowledge_search（事实约束）
  - prompt 组装
  - LLM 生成结构化回复

## 4.6 多媒体输出接口

### MediaResponder.emit
- 输入：
  - user_id: string
  - reply_plan: ReplyPlan
- 输出：EmitResult
  - sent_count: int
  - has_audio: bool
  - fallback_text_only: bool
- 行为：
  - OneSentenceChat 走 TTS。
  - SongSegmentChat 走唱段通道。
  - TTS 异常时降级纯文本发送。

## 4.7 后处理接口（异步，不计主链预算）

### PostProcessor.enqueue
- 输入：PostProcessTask
- 输出：none

### PostProcessor.run_once
- 输入：task_id
- 输出：TaskResult
- 行为：
  - 批量入库（用户消息 + Agent 回复）
  - 向量记忆写入
  - 上下文压缩
  - 人设更新计数

## 5. 时序设计

## 5.1 文本消息主链路
1. ChatStream 接收 USER_TEXT。
2. Ingress 标准化并写入 UnreadStore。
3. TopicExtractor 对快照做一次话题提取（LLM #1）。
4. 覆盖规则处理，完整话题入 TopicQueue。
5. ReplyComposer 按顺序生成回复（LLM #2）。
6. MediaResponder 执行 TTS 并发送 agent_message。
7. PostProcessor 异步执行入库/记忆/压缩/人设计数。

## 5.2 图片消息主链路
1. USER_IMAGE 进入 Ingress。
2. VLM 生成图片描述（额外调用 1 次）。
3. 与文本同路径进入 TopicExtractor 与 ReplyComposer。

## 5.3 不完整话题超时路径
1. TopicExtractor 输出 incomplete_tail。
2. 启动/重置 2 秒计时器。
3. 若期间收到 USER_TEXT 或 USER_TYPING，计时器重置。
4. 若超时，incomplete_tail 强制转完整话题并入队处理。

## 6. 容错与降级策略

1. 话题提取失败
- 降级策略：将当前 unread 快照作为单一 other 话题继续回复。
- 同时记录异常日志与追踪字段。

2. 回复生成失败
- 发送固定兜底文案，避免静默失败。

3. TTS 失败
- 发送纯文本 ChatResponse，audio 置空。

4. WebSocket 发送失败
- 记录失败并触发连接状态回收，等待重连。

5. 一致性保护
- unread 采用 snapshot + ack 模式。
- 任一阶段失败支持 rollback，避免消息丢失或重复消费。

## 9. 测试方案

1. 单元测试
- 话题切分准确性与字段完整性。
- 覆盖规则正确性（greeting 覆盖）。
- 2 秒计时器 reset 与超时触发。

2. 集成测试
- 连续多消息仅一次提取。
- 文本严格 2 次 LLM。
- 图片严格 1 次 VLM + 2 次 LLM。
- TTS 失败降级纯文本可达。

3. 并发测试
- 100 用户并发下每用户顺序一致、无串话。
- 打断与重试场景无丢消息、无重复回复。
