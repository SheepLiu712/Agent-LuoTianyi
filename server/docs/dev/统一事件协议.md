## 统一事件协议（WebSocket）

本文档用于约定前后端在 WebSocket 中交换消息的统一格式。  
目标：先保证可跑通，再逐步扩展。

---

## 1. 总体约定

所有消息都使用 JSON，对象顶层字段统一如下：

1. `type`: 事件类型，字符串，必填
2. `client_msg_id`: 客户端消息唯一 ID，字符串，可选但强烈建议
3. `reply_to`: 服务端事件关联的客户端消息 ID，字符串，服务端事件建议填写
4. `ts`: 时间戳（毫秒），数字，建议
5. `payload`: 事件内容，对象，必填

示例：

```json
{
  "type": "user_text",
  "client_msg_id": "c_20260314_0001",
  "ts": 1773474000000,
  "payload": {
    "username": "alice",
    "token": "xxx",
    "message": "你好天依"
  }
}
```

---

## 2. 客户端 -> 服务端事件

### 2.1 auth（推荐先实现，当前服务端已支持）
用于连接建立后做一次鉴权。鉴权通过后，后续业务事件不再需要每条携带 `username/token`。

`payload` 字段：
1. `username`: 用户名，必填
2. `token`: 消息 token，必填

### 2.2 user_text
用户发送文本。

`payload` 字段：
1. `message`: 文本内容，必填
2. `metadata`: 扩展字段，对象，可选

示例：

```json
{
  "type": "user_text",
  "client_msg_id": "c_20260314_0002",
  "payload": {
    "message": "你好天依",
    "metadata": {
      "scene": "chat"
    }
  }
}
```

### 2.3 user_image
用户发送图片。图片采用 `image_base64`。

`payload` 字段：
1. `username`: 用户名，必填（若已 auth 可选）
2. `token`: token，必填（若已 auth 可选）
3. `image_base64`: 图片 Base64 字符串，必填
4. `mime_type`: 图片类型，必填，例如 `image/png`、`image/jpeg`
5. `image_client_path`: 客户端图片路径，可选
6. `metadata`: 扩展字段，对象，可选

示例：

```json
{
  "type": "user_image",
  "client_msg_id": "c_20260314_img_1",
  "payload": {
    "username": "alice",
    "token": "xxx",
    "image_base64": "iVBORw0KGgoAAAANSUhEUgAA...",
    "mime_type": "image/png",
    "image_client_path": "C:/Users/A/Pictures/demo.png"
  }
}
```

### 2.4 typing
客户端输入状态事件。

`payload` 字段：
1. `is_typing`: 是否正在输入，布尔，必填
2. `username`: 用户名，可选

### 2.5 ping
心跳请求。

`payload` 字段：
1. `ping_id`: 心跳 ID，字符串，必填

---

## 3. 服务端 -> 客户端事件

### 3.1 auth_ok
鉴权成功。

### 3.2 auth_error
鉴权失败。

`payload` 字段：
1. `code`: 错误码，字符串
2. `message`: 错误信息，字符串

### 3.3 server_typing
服务端进入处理状态（可用于前端显示“正在输入”）。

### 3.4 text_chunk
文本流式分片。

`payload` 字段：
1. `text`: 文本片段，字符串，必填
2. `seq`: 分片序号，整数，必填
3. `is_final`: 是否最后一片，布尔，必填
4. `expression`: 表情，可选

### 3.5 audio_chunk
音频流式分片。

`payload` 字段：
1. `audio_base64`: 音频片段 Base64，字符串，必填
2. `audio_format`: 音频格式，字符串，必填，例如 `wav`
3. `seq`: 分片序号，整数，必填
4. `is_final`: 是否最后一片，布尔，必填

### 3.6 done
本轮回复完成事件。

### 3.7 error
处理异常。

`payload` 字段：
1. `code`: 错误码，字符串
2. `message`: 错误信息，字符串
3. `detail`: 详细信息，可选

### 3.8 pong
心跳响应。

`payload` 字段：
1. `ping_id`: 对应 ping 的 ID
2. `server_ts`: 服务端时间戳（毫秒）

---

## 4. 最小落地版本（建议先实现）

第一版只实现以下事件即可：

1. 客户端：`user_text`、`user_image`、`ping`
2. 服务端：`server_typing`、`text_chunk`、`audio_chunk`、`done`、`error`、`pong`

建议先发送 `auth`，后续 `user_text` 不再携带 `username + token`。

---

## 5. 约束与注意事项

1. `client_msg_id` 在同一用户会话内应唯一，服务端可据此去重。
2. 图片与音频都采用 Base64，注意单包大小，必要时客户端分片。
3. 服务端流式返回时，`reply_to` 应指向对应的 `client_msg_id`。
4. 发生错误时尽量返回 `error` 事件而不是直接断开连接。
5. 心跳建议 15-30 秒一次，连续超时可断开并重连。
