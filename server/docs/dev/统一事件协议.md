## 统一事件协议（WebSocket）

本文档基于当前服务端代码实现整理，面向客户端开发者。  
适用端点：`/chat_ws`。

原有的`/chat`端点和`/picture_chat`端点仍然可用，但不建议新客户端使用。
原有的`get("/history")`,`post("/get_image")`，`post("/update_image_client_path")`仍然可用，暂时不会删除，在本版本中可以继续使用。

## 1. 对接前置（登录与 token）

WebSocket 鉴权使用 `message_token`，不是 `login_token`。

1. 获取公钥：`GET /auth/public_key`
2. 用公钥加密明文密码（RSA OAEP + SHA256，Base64 编码）
3. 登录：`POST /auth/login`
4. 从返回中取 `message_token`
5. 连接 `ws://host:port/chat_ws` 或 `wss://host:port/chat_ws`，发送 `auth` 事件

登录成功返回核心字段示例：

```json
{
  "login_token": "...",
  "message_token": "...",
  "user_id": "dpon"
}
```

## 2. WebSocket 顶层消息结构

所有收发消息均为 JSON 对象，推荐统一结构：

1. `type`: 事件类型，字符串，必填
2. `payload`: 事件体，对象，建议必填（服务端会兜底为空对象）
3. `client_msg_id`: 客户端消息 ID，字符串，建议传
4. `ts`: 客户端毫秒时间戳，数字，可选
5. `reply_to`: 仅服务端事件可能携带，关联到客户端 `client_msg_id`

示例：

```json
{
  "type": "user_text",
  "client_msg_id": "c_20260318_001",
  "ts": 1773474000000,
  "payload": {
    "message": "你好，天依"
  }
}
```

## 3. 连接与鉴权时序

1. 建立连接后，服务端会先下发 `system_ready`
2. 客户端发送 `auth`
3. 成功收到 `auth_ok` 后，再发送聊天与心跳事件

时序示例：

```text
S -> C: system_ready
C -> S: auth
S -> C: auth_ok
C -> S: hb_ping
S -> C: hb_pong
C -> S: user_text
S -> C: server_ack (reply_to=user_text.client_msg_id)
S -> C: agent_state_changed (listening/thinking/...)
S -> C: agent_message
```

## 4. 客户端 -> 服务端事件

### 4.1 auth

用途：会话鉴权。

`payload` 字段：
1. `username`: 字符串，必填
2. `token`: 字符串，必填（即 `message_token`）

示例：

```json
{
  "type": "auth",
  "client_msg_id": "auth-1",
  "payload": {
    "username": "dpon",
    "token": "<message_token>"
  }
}
```

### 4.2 hb_ping

用途：心跳保活。

`payload` 字段：
1. `ping_id`: 整数，必填（建议严格递增）

示例：

```json
{
  "type": "hb_ping",
  "client_msg_id": "ping-12",
  "payload": {
    "ping_id": 12
  }
}
```

注意：
1. `ping_id` 缺失时服务端返回 `error`，`code=MISSING_PING_ID`
2. 服务端仅在 `ping_id` 递增时返回 `hb_pong`

### 4.3 user_text

用途：发送文本消息。

`payload` 字段（至少一个文本字段非空）：
1. `message`: 字符串，推荐
2. `text`: 字符串，可选
3. `content`: 字符串，可选

服务端按 `message -> text -> content` 顺序取第一条非空字符串。

示例：

```json
{
  "type": "user_text",
  "client_msg_id": "msg-1",
  "payload": {
    "message": "你今天心情怎么样？"
  }
}
```

### 4.4 user_image

用途：发送图片语义输入（当前实现会转成“用户发送了一张图片”的事件语义）。

`payload` 字段：当前服务端未强校验具体字段，建议包含：
1. `image_base64`: 字符串
2. `mime_type`: 字符串
3. `image_client_path`: 字符串，可选

### 4.5 user_typing

用途：告知“正在输入”，用于延长 listening 阶段等待时间。当输入框内消息更改时，前端可以持续发送 `user_typing` 事件。发送间隔建议为0.5s到1s之间。

`payload` 字段：当前服务端不读取具体字段，可传：
1. `is_typing`: 布尔

注意：事件名必须是 `user_typing`，不是 `typing`。

### 4.6 兼容别名事件

服务端目前还接受以下文本类别名事件并按聊天消息处理：
1. `user_message`
2. `message`
3. `chat_message`
4. `chat`

建议新客户端统一使用 `user_text`。

## 5. 服务端 -> 客户端事件

### 5.1 system_ready

用途：连接建立后提示客户端先鉴权。

`payload` 示例：

```json
{
  "message": "WebSocket connected. Please send auth first.",
  "require_auth_before_chat": true
}
```

### 5.2 auth_ok

用途：鉴权成功。

`payload` 示例：

```json
{
  "message": "authentication successful for user dpon"
}
```

通常会带 `reply_to`，值为对应 `auth` 的 `client_msg_id`。

### 5.3 auth_error

用途：鉴权失败。

`payload` 字段：
1. `code`: 错误码
2. `message`: 错误信息

当前错误码：
1. `MISSING_AUTH_FIELDS`
2. `INVALID_TOKEN`

### 5.4 hb_pong

用途：心跳响应。

`payload` 字段：
1. `ping_id`: 与请求对应
2. `server_ts`: 服务端毫秒时间戳

### 5.5 agent_state_changed

用途：Agent 状态变化通知，可用于前端状态显示。

`payload` 字段：
1. `state`: 字符串

当前可能值：
1. `waiting`
2. `reflection`
3. `listening`
4. `thinking`

### 5.6 agent_message

用途：Agent 回复内容包（文本/音频/表情）。

`payload` 对应服务端 `ChatResponse`：
1. `uuid`: 字符串
2. `text`: 字符串
3. `audio`: 字符串或 `null`（通常为 Base64 音频数据，当前简化链路可能为空字符串）
4. `expression`: 字符串或 `null`
5. `is_final_package`: 布尔

示例：

```json
{
  "type": "agent_message",
  "ts": 1773474001111,
  "payload": {
    "uuid": "b4f9ec0c-5f7d-40e9-83eb-7cb1f58d52f1",
    "text": "用户输入：你好",
    "audio": "",
    "expression": "微笑脸",
    "is_final_package": true
  }
}
```

### 5.7 error

用途：协议错误或消息格式错误。

`payload` 字段：
1. `code`
2. `message`

当前常见错误码：
1. `BAD_JSON`
2. `BAD_MESSAGE`
3. `MISSING_PING_ID`

### 5.8 server_ack

用途：确认服务端已接收并入队一个客户端聊天输入事件。

当前会发送 ACK 的客户端事件类型：
1. `user_text`
2. `user_image`
3. `user_typing`

`payload` 字段：
1. `received_event_type`: 字符串，表示被确认接收的事件类型

通常会带 `reply_to`，值为对应客户端事件的 `client_msg_id`。

示例：

```json
{
  "type": "server_ack",
  "ts": 1773474001000,
  "reply_to": "msg-1",
  "payload": {
    "received_event_type": "user_text"
  }
}
```

## 6. 关键行为约束（按当前实现）

1. 鉴权前发送聊天消息不会进入聊天流程；必须先 `auth` 成功。
2. 非 JSON、非对象、缺少 `type` 的消息会收到 `error`。
3. 当前已实现接收确认事件 `server_ack`，但未实现重传与 `done` 事件；客户端应以 `agent_message` 和 `agent_state_changed` 作为主要渲染依据。
4. 断线后可重连，同一用户会复用对应 chat_stream（服务端侧管理）。
5. 心跳建议每 5 到 30 秒发送一次，避免连接被服务端清理任务回收。

## 7. 客户端最小实现清单

1. 实现登录取 `message_token`
2. 连接 `/chat_ws`
3. 收到 `system_ready` 后发送 `auth`
4. 收到 `auth_ok` 后启动心跳 `hb_ping`
5. 发送 `user_text`
6. 监听并处理 `agent_state_changed` 与 `agent_message`
7. 统一处理 `error`/`auth_error`
