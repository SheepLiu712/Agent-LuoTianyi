# 架构更新需求
## 项目介绍
见README.md

## 问题背景
### 现有问题
#### 用户体验方面
1. 用户每发送一条信息，必须等待天依完全回复后才能继续发送下一条信息，导致交流效率低下。用户希望能够发送任意条信息，并且天依能够同时处理多条信息，提升交流效率。
2. 天依存在复读现象，即在某些情况下会重复回复相同的信息。这可能是记忆库反复抓取到同一条信息导致的。

#### 技术方面
1. 目前的是按照一问一答处理的，连接必须在天依处理信息时保持畅通，否则会导致信息丢失或处理失败。
2. 目前回复需要至少通过3次LLM调用（图片是4次），效率较低。
3. 目前的架构设计较为复杂和混乱，维护和扩展较为困难。

### 本次更新需要实现的功能
1. 改用websocket连接;
2. 改用异步处理，天依不必然在用户发送信息后立刻生成回复；
3. 改进系统架构，尽量降低回复延时。确认延时主要来自于LLM还是TTS。
4. 优化长期记忆、术语的存储和检索，减少复读现象，提高回复的多样性和准确性。

### 预期目标
1. 用户在发送信息后，等待的时间合适。天依在合适的时间开始生成回复（客户端将显示“正在输入”），并在可接受的时间内（如2-5秒）开始给出回复（近似于首字延时）。回复不仅包括文本，也需要同时生成TTS音频。
2. 天依的回复内容丰富且多样化，避免复读现象。
3. 对于圈子内的用户，天依能够准确检索到歌曲、活动信息等相关信息，并在回复中合理使用这些信息。
4. 长期而言，天依能够记住①用户的身份与爱好；②用户说过的话；
5. 为天依开发好感度和日程系统，并在回复中合理使用这些信息。

### 关键技术点
1. websocket连接的实现和管理；
2. 异步处理的设计和实现；
3. 系统架构的优化，降低回复延时（可能是降低LLM调用数量，或使用流式输出）；
4. 长期记忆和术语的优化存储和检索机制设计与实现。
5. context竞争覆盖机制，使得LLM不必面对极大量的，对聊天回复没有帮助的信息。

## 实施方案
### 敏捷开发优先级
1. 首先实现异步处理和websocket连接，确保用户能够同时发送多条信息，并且天依能够同时处理多条信息。
2. 在不考虑任何长期记忆、术语等额外信息的情况下，跑通回复流程，确认回复的延时主要来自于LLM还是TTS。在此基础上对架构进行重新设计，设置合理的抽象。
3. 考虑术语、长期记忆、用户画像的存储和检索机制，优化系统架构，降低回复延时。考虑context竞争覆盖机制。
4. 加入好感度和日程系统，并在回复中合理使用这些信息，同样需要优化系统架构，降低回复延时。

### 可能的流程
1. 用户通过websocket连接发送信息，服务器接收信息并将其放入消息队列中。
2. 即刻从消息中提取术语（梗和黑话），进行翻译。这可以通过关键词匹配或使用专门的术语识别模型来实现。flashtext或者jieba等工具可以用来实现高效的术语提取。
3. 快速决策是等待用户的下一句，还是即刻开始生成回复。同时决策回复所需的知识。决策可以用小模型或者用LLM的API。
4. 如果确定要回复，立即开始生成回复。回复的类型包括文本（后接TTS合成）以及文本+歌曲片段（本地存储的音频文件）。
5. 后处理：包括调整用户画像、好感度、写入长期记忆、写入新的术语等。

### 可能的技术栈
- websocket连接：可以使用Python的`websockets`库或者`FastAPI`的WebSocket支持来实现服务器端的WebSocket连接。
- 术语提取：可以使用flashtext或者jieba等工具来实现高效的术语提取。
- 快速决策：可以使用一个本地的小模型或者轻量的API调用来进行快速决策，确定是否开始生成回复以及需要哪些知识。
- context ranking：考虑通过算法来评估当前对话上下文中哪些信息对生成回复最有帮助，从而减少LLM需要处理的信息量，降低延时。
- 回复生成：可以继续使用现有的LLM模型来生成回复，但需要优化调用方式，可能需要使用流式输出或者减少调用次数来降低延时。
