# 记忆系统更新需求
我们需要对src/memory模块进行一些更新，以提高记忆保存和检索的准确性。

## 架构说明
由memory_manager.py负责管理整个向量记忆模块，提供接口供外部调用。向量记忆模块由以下几个部分组成：
- VectorStore：负责存储和管理向量数据，提供插入、查询等功能，它目前在database/vector_store.py中实现。
- MemorySearcher：负责根据查询内容在VectorStore中检索相关记忆，目前在memory_searcher.py中实现。
- MemoryWriter：负责将新的记忆数据写入VectorStore，目前在memory_writer.py中实现。

目前，所有用户的记忆都写在同一个VectorStore中通过user_id在查询时进行隔离。我们将保持这个架构。

目前，通过定义工具的方式，让LLM能够调用工具来查询和写入记忆。我们将继续使用这种方式。

## 架构问题
现在的架构有以下问题：
1. 我们曾经尝试避免反复查询记忆，因此提供了“继承上一次记忆”的功能。但这会导致记忆检索越来越多，最终导致检索效率降低。
2. 我们提供了更新记忆的功能，但LLM会滥用这个功能导致记忆越来越长。
3. 我们尝试让LLM避免反复写入同样的记忆，实现方法是把最近写入的记忆放入上下文。但这不解决问题，因为新的记忆可能和更早之前的记忆重复。
4. 我们没有明确LLM要记什么，导致它会记一些鸡毛蒜皮的小事。
5. 我们没有明确记忆的格式，导致LLM写入的记忆格式混乱，检索命中率降低。

## 更新需求
针对上述问题，我们需要对记忆系统进行以下更新：
1. 明确记忆的内容：我们要求记忆只记以下内容
   1. 用户的兴趣爱好、习惯、偏好等长期信息。例如，用户喜欢什么类型的音乐、饮食，讨厌什么，平时有什么习惯等。
   2. 用户分享的生活事件、经历等信息。例如用户说“我今天去了公园，玩得很开心。”
2. 明确记忆的格式，我们需要显式要求记忆足够简明，最好控制在一句话内。同时文档保存时要保存记忆产生的时间。
3. 记忆写入。记忆写入的时间应该在每一个Topic的回复生成之后。明确提供两个接口写入不同类型的记忆：
   1. write_user_memory(user_id: str, content: str)：写入用户相关的记忆，content应该包含用户的兴趣爱好、习惯、偏好等长期信息。
   2. write_event_memory(user_id: str, content: str)：写入事件相关的记忆，content应该包含用户分享的生活事件、经历等信息。
   3. 在每次写入记忆时，对于user_memory，检查是否有相似记忆，有则不更新；
   4. 对于event_memory，如果日期不同直接写入，不去重。
4. 记忆检索。检索记忆时暂时将这两种记忆混起来检索，取相关度最高的k条，并且要求分数大于thres。

## 更新内容
1. 梳理memory模块的结构。
2. 重新写memory_search和memory_write工具的prompt，明确记忆的内容和格式要求。
3. 重新设计memory_manager的接口，并在luotianyiagent中更简洁地调用这个接口。
4. 在topic_replier中调用新的记忆写入接口（作为一个asyncio任务），在每次生成回复后写入相关的记忆。
5. 检查topic_replier中调用记忆检索的部分，确保它能够正确地调用新的检索接口，并且正确处理返回的记忆结果。
