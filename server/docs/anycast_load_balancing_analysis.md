# Anycast 负载均衡方案分析

## 背景

洛天依 Agent 是一个实时聊天 + TTS 语音合成的 WebSocket 应用，需要将用户流量分发到多个服务端实例以实现横向扩展和高可用。

## 什么是 Anycast

Anycast 是一种网络路由技术：**多个地理分散的服务器共享同一个 IP 地址**。BGP 路由协议将用户导向"最近"的服务器节点（按 AS 跳数度量）。

```
用户 A (北京) ──→ 北京节点 (同一个 IP)
用户 B (广州) ──→ 广州节点 (同一个 IP)
用户 C (成都) ──→ 成都节点 (同一个 IP)

所有用户看到的是同一个 IP：203.0.113.1
```

## Anycast 对本项目的适用性分析

### 优势

| 方面 | 评价 |
|------|------|
| **客户端零配置** | 客户端只需要一个 IP，无需配置多个地址 |
| **自动故障切换** | 节点宕机后 BGP 自动收敛，流量切到其他节点 |
| **DDoS 缓解** | 流量自然分散到多个节点，单点压力降低 |
| **就近接入** | 用户自动连接到延迟最低的节点 |

### 劣势

| 方面 | 评价 |
|------|------|
| **WebSocket 有状态问题** | 长连接不通过 Anycast 迁移，断连后需重连，重连可能落到不同节点 |
| **TTS 状态共享复杂** | TTS 音频生成结果存储在本地，其他节点无法获取 |
| **GPU 资源分散** | 每个节点都需要独立 GPU，无法集中利用 |
| **BGP 运维成本高** | 需要 ASN、IP 段、BGP 对等互联，个人/小团队难以维护 |
| **成本高昂** | 需要多台服务器，每台都需要 GPU |

### 关键限制

#### 1. WebSocket 连接迁移

本项目使用 WebSocket 长连接进行聊天。Anycast 路由在 TCP 连接建立后不会动态切换。如果北京节点宕机：

```
连接断开 → 客户端重连 → DNS/TCP 路由到广州节点
                         ↓
                  用户在广州节点的 Redis 中无最近上下文
                  需要从 SQL 重新加载
                  前面几轮对话的 TTS 音频不可用
```

**缓解方案：** 所有节点共享 Redis + SQL，但 TTS 本地文件需要网络存储（NFS/S3）。

#### 2. TTS 串行化瓶颈

当前架构使用 `GlobalSpeakingWorker` **全局串行化** TTS 任务以防止 GPU OOM。跨节点时：

- 如果 TTS 分布到多个 GPU，需要分布式锁（如 Redis Lock）
- 串行化的优势被打破，多个 GPU 可能同时推理
- 要么接受 GPU 可能 OOM，要么引入分布式协调（复杂度↑）

#### 3. 状态粘性

用户偏好、LLM 端点配置等存储在本地 SQL 和 Redis。Anycast 要求所有节点共享存储：

- SQL 已共享（同一个 SQLite 不能跨节点，需升级为 PostgreSQL）
- Redis 已共享（需改为同一集群）
- TTS 参考音频 / 本地文件需要 NFS 或对象存储

### 评分矩阵（1-5 分，5 最佳）

| 维度 | Anycast | 反向代理 (Nginx) | DNS 负载均衡 | 说明 |
|------|---------|-------------------|-------------|------|
| 实现复杂度 | ⚠️ 2 | ✅ 4 | ✅ 5 | Anycast 需要 BGP 运维 |
| 成本 | ❌ 1 | ✅ 4 | ✅ 5 | Anycast 需要多台服务器 |
| 故障切换速度 | ✅ 5 | ✅ 4 | ⚠️ 2 | BGP 收敛快于 DNS TTL |
| 客户端配置 | ✅ 5 | ✅ 5 | ✅ 5 | 都是单地址 |
| 会话保持 | ❌ 2 | ✅ 4 | ⚠️ 2 | Nginx sticky sessions 更可靠 |
| WebSocket 支持 | ⚠️ 3 | ✅ 5 | ⚠️ 3 | Nginx 原生支持 WebSocket |
| DDoS 防护 | ✅ 5 | ⚠️ 3 | ⚠️ 3 | Anycast 天然分散 |
| GPU 利用效率 | ❌ 2 | ✅ 4 | ✅ 4 | 集中负载均衡更好分配 GPU |

## 推荐的负载均衡方案

### 方案 A：Nginx 反向代理（推荐 ✅）

```
用户 ──→ Nginx (端口转发/SSL 终止)
          ├── 轮询 / least_conn → Node 1 (GPU)
          ├── sticky session  → Node 2 (GPU)
          └── 健康检查自动摘除故障节点

所有节点共享：
  - SQL (迁移到 PostgreSQL)
  - Redis 集群
  - NFS / S3 文件存储
```

**优点：**
- 实现简单，运维成本低
- Sticky session 保证 WebSocket 不中断
- 可以按 least_conn 动态分配负载
- SSL termination 统一处理
- GPU 利用率更高（可以按需扩展节点数）

**缺点：**
- 单点故障（可加备用 Nginx + Keepalived 虚 IP）
- 需要横向扩缩容时手动配置

### 方案 B：Kubernetes + 内网穿透（推荐用于多节点扩展）

```
用户 ──→ SakuraFrp / Ngrok ──→ K8s Ingress ──→ Pod 1 (GPU)
                                          └── Pod 2 (GPU)
```

**适用场景：** 同一机房有多台 GPU 机器。

### 方案 C：Anycast（仅限专业运维团队）

```
条件：
  - 有自己 ASN 和 IP 段
  - 至少 3 个地理位置的数据中心
  - 每个数据中心都有 GPU 服务器
  - 共享存储 (PostgreSQL + Redis + S3)

适用场景：
  - 面向全国用户的低延迟需求
  - 需要 DDoS 防护
  - 团队有专业网络运维能力
```

## 针对本项目现阶段建议

1. **目前（单机部署）**：不需要负载均衡。用 `setup_server.py` 一键部署后配合 SakuraFrp 暴露内网地址即可。

2. **用户量增长后**：使用 **Nginx 反向代理 + sticky sessions**，多个 GPU 节点在同一内网。

3. **大规模运营**：
   - SQLite → PostgreSQL 迁移（必需）
   - 引入 Redis Cluster（已有 Redis，只需配置集群模式）
   - 文件中转 → S3/MinIO（TTS 音频等）
   - 考虑 Anycast 或全局负载均衡（GSLB）

## 结论

| 方案 | 适合场景 | 推荐度 |
|------|---------|--------|
| **Anycast** | 多数据中心、专业运维团队、需要 DDoS 防护 | ⭐⭐ |
| **Nginx 反向代理** | 单机房多节点、团队规模小、快速实现 | ⭐⭐⭐⭐⭐ |
| **K8s + Ingress** | 容器化部署、自动扩缩容、已有 K8s 基础设施 | ⭐⭐⭐ |
| **DNS 轮询** | 无需粘性的场景、简单的读写分离 | ⭐⭐ |

**Anycast 不适合本项目当前的阶段。** 优先推荐 Nginx 反向代理方案——实现成本低、运维简单、足够满足 WebSocket + TTS 场景的负载均衡需求。只有当用户量级达到月活数万以上，且需要多地域就近接入时，才值得考虑 Anycast。
